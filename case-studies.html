1. The Wi-Fi Toggle Switch (Binary Choice Widget)

This case study examines a simple interaction on a modern mobile device (e.g., iPhone or Samsung) using parallel models to understand user intent and system response.

Norman's Action Cycle Analysis:

  • Goal: The user recognizes a need for internet access: "I need Wi-Fi ON".
  • Plan: A mental simulation occurs where the user decides to use a specific method, such as the "quick toggle" menu for efficiency rather than going through deep system settings.
  • Specify: The user performs a visual search to locate the fan-shaped Wi-Fi icon among other toggles.
  • Execute: The physical motor action takes place; for example, a finger taps the toggle on a touchscreen (a process roughly estimated at 0.2 seconds).
  • Perceive & Interpret: The user sees the switch move and the color change from grey to blue, interpreted as "Wi-Fi is now ON".
  • Evaluate: The user verifies the outcome by looking for the Wi-Fi icon in the status bar or attempting to load a webpage.

GOMS & KLM Interpretation:

  • Goal/Method: The top-level objective is "Enable-Wi-Fi". The GOMS "Method" corresponds to Norman's "Plan" (e.g., Swipe down → Control Center → Wi-Fi).
  • Operators (KLM): GOMS abstracts the cognitive "Specify" stage, focusing on the timed motor execution. The KLM sequence is modeled as [M] (Mental preparation, ~1.35s), [P] (Pointing to the icon, ~1.1s), and [K] or [B] (Keystroke or Button tap, ~0.2s).
  • Divergence: Norman's model focuses on closing the Gulf of Evaluation (Perceive/Interpret/Evaluate), which is entirely outside the scope of GOMS, as GOMS assumes perfect perception and instant goal satisfaction.

2. The Automatic Syringe (Dose Calculator)

This case study highlights the difference between a "recall-heavy" interface and one designed for "recognition," focusing on safety-critical errors.

Bad Design (Numeric Keypad):

  • Analysis: The user must enter doses via a numeric keypad. This forces an extra abstraction layer: Clinical intent → Abstract numeric encoding → Motor execution.
  • Gulf of Execution: The user must recall the exact digit sequence (e.g., "0-1-2-.-5-0" for 12.5 ml/hr). This heavy memory load leads to a wide Gulf of Execution.
  • Gulf of Evaluation: If the display shows only "1250," it is ambiguous; the user must mentally convert the number to determine if it means 12.5 or 125.0, resulting in a wide Gulf of Evaluation.
  • Consequence: A single key slip (a "slip" in execution) can cause a 10x overdose (e.g., setting 1372 instead of 137).

Good Design (Individual Digit Buttons):

  • Analysis: The redesigned interface uses individual increment/decrement buttons for each digit.
  • Norman's Stages: The "Plan" shifts from recall to visual adjustment (recognition).
  • Specify/Execute: The user specifies the action by pressing larger, error-resistant buttons for specific units (Tens, Units, Tenths), reducing motor complexity and following Fitts's Law more effectively.
  • Perceive/Evaluate: The display clearly shows "12.50 ml" with units, providing a direct visual mapping that narrows the Gulf of Evaluation and increases confidence in accuracy.

References: Dix et al., HCI (Introduction, p. 8); Week 02 Lecture Slides; Week 03 Quiz Worksheet; Week 04 Lab Material.

3. The "Who Paid?" Problem (Splitwise Analysis)

This study treats an interaction as a social coordination system where design decisions have emotional and social consequences.

Norman's Action Cycle Analysis:

  • Goal: A complex intent involving social fairness: "Record that Maria paid 5000 PKR so everyone pays fairly".
  • Plan & Specify: The user must mentally plan multiple steps: open the app, enter a description, and select the payer and participants. This involves a high cognitive load because the user must recall who ordered what and individual prices.
  • Execute: The physical actions—typing, selecting, and checking—often interrupt social conversation at a dinner table.
  • Perceive & Interpret: The interface must provide multiple feedback channels (e.g., green checkmark animation, "Expense added!" text, and notification sounds sent to others).
  • Evaluate: The user verifies the action by checking the group history and confirming that others have acknowledged receipt.

Social and Cognitive Challenges:

  • Social Trust: Evaluation is not just about data entry; it is about social trust (e.g., "Did everyone get notified?") to avoid arguments about money among friends.
  • Social Discomfort: Explicit monetary calculations in groups can increase "calculating" behavior, causing social awkwardness or the fear of being seen as "cheap".
  • Outcome Visibility: Ambiguity in outcome visibility (not knowing immediately who owes what) leads to manual reasoning and potential disputes.
